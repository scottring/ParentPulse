# Therapy ŌĆö Design

**Date:** 2026-04-16
**Status:** Approved for planning
**Scope:** A new top-level publication ŌĆö **Therapy** ŌĆö alongside Journal and Manual. Helps the user prepare talking points for upcoming therapy sessions by clustering activity since the last session into themes, tracks which themes get discussed, carries undiscussed threads forward, and imports self-authored session notes / transcripts.

---

## Motivation

The user wants "an elegant way to prepare talking points for therapy sessions based on activity since the last therapy session." This slots directly into the project's coherent story: *Relish is a journal that other people help you write, and that writes you back.* Therapy prep is one of the highest-leverage forms of "writes you back" ŌĆö the app synthesizes a week of entries + marginalia + manual signals into a small number of themes that are worth raising with a therapist.

Therapy is promoted to a third top-level book (alongside Journal and Manual) because it has the properties of a genuine chronicle: its own cadence (per-session windows), its own reader (the therapist), its own longitudinal arc (unresolved threads, recurring themes across months), and its own ritual (session-close starts a new chapter). Growth items were demoted because they were atomic leaves with no narrative; Therapy is not that ŌĆö it accumulates.

v1 targets **personal individual therapy** for one user. The data model anticipates **couples** and **family** therapy as future extensions (a `kind` field on the therapist entity is the hinge), but none of that is built in v1.

## Scope

**In scope**

- New top-level route `/therapy` and child routes `/therapy/sessions/[windowId]`, PIN-gated like `/journal`.
- `therapists` entity ŌĆö v1 creates exactly one, `kind: 'individual'`.
- `therapy_windows` ŌĆö the time span between two sessions (one open + many closed).
- `therapy_themes` ŌĆö 2ŌĆō4 clusters per open window, generated by Claude, each with 1ŌĆō3 source refs linking back to source material; supports star / dismiss / per-theme note.
- `therapy_notes` ŌĆö self-authored session records, paste-in only, up to full transcripts.
- Regen Cloud Function with three invocation paths: daily cron, on-demand refresh, session-close.
- "I had a session" flow: mark themes discussed, optionally paste transcript, close window, open a fresh window with undiscussed themes carried forward.
- Surface card that promotes the workspace near session time without leaking content.
- Route-level PIN gating + Firestore rules restricting all four collections to `ownerUserId`.

**Out of scope**

- Couples and family therapy. The `kind` field and forward-compatible shape exist; behavior is not built.
- Calendar integration ("my therapist is every Thursday"). Session close is explicit-button-only in v1.
- File upload / PDF / OCR of therapist-shared notes. Paste text only.
- Email-forward import.
- Reordering themes by drag.
- PDF export, share links, or any outbound sharing of prep content with a therapist.
- Search across past sessions.
- AI-generated questions for the therapist to ask (Q5 option D ŌĆö deferred).
- Per-theme reactions / emoji.

## Decisions locked with the user

1. **Who it's for:** Personal individual therapy v1; architected so couples/family extend without migration. (Q1.)
2. **Shape:** Hybrid ŌĆö a running workspace draft maintained in the background, experienced as a prep view at session time. (Q2 option C.)
3. **Session marker:** Explicit "I had a session" button. No calendar integration in v1. (Q3 option B.)
4. **Input corpus:** Journal entries (all visibility, including "just me"), marginalia on those entries, manual synthesis deltas / growth items / dimension shifts, and the previous window's `therapy_notes` as historical context. No per-source toggles. (Q4 option C.)
5. **Output shape:** 2ŌĆō4 themed clusters, each with a 1ŌĆō3 supporting sources and a `carriedForward` pip when applicable. (Q5 option B.)
6. **Regen cadence:** Daily cron + on-demand refresh + session-close, with last-updated timestamp surfaced if stale. (Q6 option D.)
7. **Location in app:** Third top-level book at `/therapy`, promoted on The Surface by a count-only card. (Q7 ŌĆö final, after initial confusion about the current nav state.)
8. **Session-close ritual:** Checkbox per theme to mark discussed; undiscussed themes carry forward into the new window with `lifecycle.carriedForwardCount` incremented. (Q8 option C.)
9. **Theme interactions between sessions:** Star, dismiss, per-theme note. No reorder. (Q9 option B.)
10. **Privacy posture:** Always PIN-gated; The Surface hides the card entirely while PIN-locked; Surface card shows counts only, never theme content. (Q10 option C.)
11. **Note import:** Self-authored only (transcripts included), paste text only, attaches to the window it documents. (Q11 options A + i.)

## Architecture

### Data model ŌĆö four new Firestore collections

All four collections live at the top level, all owned by `ownerUserId`, all PIN-gated at the route level and rule-level restricted to the owner.

```ts
// src/types/therapy.ts (new)
import { Timestamp } from 'firebase/firestore';

export type TherapistKind = 'individual' | 'couples' | 'family';

export interface Therapist {
  id: string;
  ownerUserId: string;
  displayName: string;
  kind: TherapistKind;           // v1 always 'individual'
  createdAt: Timestamp;
}

export interface TherapyWindow {
  id: string;
  therapistId: string;
  ownerUserId: string;
  status: 'open' | 'closed';
  openedAt: Timestamp; 
  closedAt?: Timestamp;
  themeIds: string[];              // current themes in this window
  noteIds: string[];               // therapy_notes attached
  carriedForwardFromWindowId?: string;
  lastRegeneratedAt?: Timestamp;
}

export type TherapySourceKind =
  | 'entry'
  | 'marginalia'
  | 'synthesis'
  | 'growth_item'
  | 'therapy_note';

export interface TherapySourceRef {
  kind: TherapySourceKind;
  id: string;
  snippet: string;                 // first ~200 chars, for workspace render without refetch
}

export interface TherapyTheme {
  id: string;
  windowId: string;
  therapistId: string;
  ownerUserId: string;
  title: string;
  summary: string;
  sourceRefs: TherapySourceRef[];

  userState: {
    starred: boolean;
    dismissed: boolean;
    note?: string;
  };

  lifecycle: {
    firstSeenWindowId: string;
    carriedForwardCount: number;
    discussedAt?: Timestamp;
  };

  generatedAt: Timestamp;
  model: string;                   // the Claude model id used
}

export interface TherapyNote {
  id: string;
  windowId: string;
  therapistId: string;
  ownerUserId: string;
  content: string;                 // paste-in text, up to ~200k chars (Firestore 1MB doc cap)
  createdAt: Timestamp;
  updatedAt?: Timestamp;
}
```

**Why themes are their own docs** (not embedded in the window): themes need to migrate across windows when carried forward, keeping their `lifecycle` (especially `firstSeenWindowId` and `carriedForwardCount`) intact. Embedding in the window would force us to copy and reassign IDs, losing continuity. `userState` also gets written frequently (every star/dismiss/note edit), and a per-theme doc avoids rewriting the whole window on every tap.

**Why source refs store snippets**: the workspace must render theme cards with attribution ("From 4 moments") without n+1 fetches. The snippet is enough to render; tapping expands to the real source. If a source doc is later deleted, the snippet renders as "(moment removed)" ŌĆö no error.

### Firestore security rules

All four collections follow the same shape:

- **Create / Update / Delete / Read:** `request.auth.uid == resource.data.ownerUserId` (or `request.resource.data.ownerUserId` on create).
- **No family/observer read paths.** Therapy content is strictly owner-only.
- **Update constraints on `therapy_windows`:** `ownerUserId` and `therapistId` are immutable; `status` transitions only `open ŌåÆ closed`, never reverse; `closedAt` writable only when transitioning to closed.
- **Update constraints on `therapy_themes`:** clients may write `userState` only; `lifecycle`, `windowId`, `sourceRefs`, `summary`, `title`, `generatedAt`, and `model` are function-only (written by Cloud Functions with admin credentials, bypassing rules).
- **Session-close state changes are function-only.** The client calls `closeTherapyWindow` (see below); it performs the window transition, carry-forward `windowId` + `lifecycle` updates on themes, and `discussedAt` writes on discussed themes, all server-side. The only client-writable mutation on themes is `userState`.

Indexes needed:
- `therapy_windows` by `(therapistId, status)` ŌĆö "the open window for this therapist."
- `therapy_windows` by `(therapistId, closedAt desc)` ŌĆö past sessions list.
- `therapy_themes` by `(windowId, userState.dismissed asc, userState.starred desc)` ŌĆö workspace ordering.
- `therapy_notes` by `(windowId, createdAt desc)`.

### Cloud Functions

Two HTTPS callables + one scheduled trigger.

#### `regenerateTherapyWindow`

**Callable params:** `{ windowId }`. Resolves therapist + owner; verifies caller is owner.

**Scheduled trigger:** daily at a fixed hour, iterates `therapy_windows` where `status == 'open'` and `lastRegeneratedAt` is either unset or older than ~20h.

**Input gathering (scoped to `[window.openedAt, now()]`, ownerUserId):**
- `entries` / `journal_entries` authored by the owner in the range (all visibility; private "just me" entries are included).
- `margin_notes` on those entries.
- `person_manuals` synthesis where `updatedAt` falls in the range, plus any growth items newly surfaced.
- `therapy_notes` from the *previous* closed window (via `carriedForwardFromWindowId` on the current window), passed as "already-discussed context."

**Generation (Claude Sonnet via callable pattern matching `synthesizeManualContent`):**
- Prompt accepts: source material, existing open themes (with `userState` and `lifecycle`), and prior-window notes.
- Prompt instructs: return 2ŌĆō4 themes. For each, include `title`, `summary`, `sourceRefs[]`, and an `identity` field ŌĆö either `existingThemeId` (this continues an existing theme) or `null` (new).
- Dismissed existing themes are passed in with instruction "user has marked these as not useful ŌĆö do not resurface the same angle."
- Starred existing themes are passed in with instruction "user wants to raise these ŌĆö preserve them."
- Carried-forward themes are marked "not yet discussed ŌĆö preserve lifecycle."

**Write phase (single Firestore batch):**
- Returned theme with `existingThemeId`: update `title`, `summary`, `sourceRefs`, `generatedAt`, `model`. Leave `userState` and `lifecycle` untouched.
- Returned theme with `identity: null`: create doc with `lifecycle.firstSeenWindowId = windowId`, `lifecycle.carriedForwardCount = 0`.
- Existing theme not returned: keep the doc, remove its id from `window.themeIds` (soft-retire ŌĆö doc stays for debug, disappears from UI).
- Update `window.themeIds` and `window.lastRegeneratedAt`.

**Failure mode:** if the LLM call fails or returns malformed JSON, leave the window's themes untouched and log. Client UI shows "last updated Xh ago" so stale state is legible.

**Cost note:** one Claude Sonnet call per open window per day; at v1 scale (one user, one therapist), roughly $0.01/day. On-demand refreshes are additive but user-initiated.

#### `closeTherapyWindow`

Callable invoked by `SessionCloseSheet`. Encapsulates the whole session-close transition on the server side so theme `lifecycle` and `windowId` stay function-only.

**Params:** `{ windowId, sessionDate, discussedThemeIds: string[], transcript?: string }`.

**Flow (single Firestore batch):**
- Verify caller owns `windowId`'s therapist and that the window is currently `open`.
- For each theme id in `discussedThemeIds`: set `lifecycle.discussedAt = sessionDate`.
- For each theme currently in `window.themeIds` but *not* in `discussedThemeIds`: reassign `windowId` to the new window (created below) and increment `lifecycle.carriedForwardCount`.
- Transition the current window: `status = 'closed'`, `closedAt = sessionDate`.
- If `transcript` present: create `therapy_notes/{id}` attached to the closing window.
- Create a new `therapy_windows/{id}` with `status: 'open'`, `openedAt: sessionDate`, `carriedForwardFromWindowId: <closed id>`, `themeIds: [<undiscussed ids>]`.
- After batch commits, synchronously invoke `regenerateTherapyWindow` logic on the new window (shared internal handler ŌĆö not a separate callable round-trip) so the client gets a fresh workspace in one call.

**Failure modes:** double-close is caught by the "must be open" check; malformed `discussedThemeIds` (ids that don't belong to the window) are ignored with a warning. Batch either fully commits or fully fails.

### UX detail

**`/therapy` layout** ŌĆö single column, prep-brief feel, not a magazine spread. Matches Journal's chrome (wordmark header, PIN feel) but uses a tighter, more utilitarian type scale ŌĆö this is a working document, not a reading experience.

Top to bottom inside the open window:

1. **Window header** ŌĆö "This session's prep ┬Ę opened Apr 9 ┬Ę 7 days in." Right-side actions: Refresh ┬Ę Import notes ┬Ę I had a session.
2. **Carry-forward banner** ŌĆö conditional. "2 threads carried forward from your last session." Small, muted. Themes with `carriedForwardCount > 0` get a `carried` pip on their card.
3. **Theme cards** ŌĆö starred first, then LLM order, dismissed hidden. Each card: title (1 line), summary (2ŌĆō3 lines), 3 icon actions (star, dismiss, note), and a `From N moments` row that expands to snippets linking to the source entry / marginalia / synthesis / growth item / prior note.
4. **Past sessions list** ŌĆö collapsed, "Past sessions ┬Ę 12"; tap to expand.

**Session-close sheet** ŌĆö checkbox per open theme (default unchecked), "select all" / "select none", session date field defaulting to today, optional paste-transcript textarea, Confirm. Copy for the all-unchecked case: "Nothing discussed? That's fine ŌĆö everything carries forward."

**Import notes modal** ŌĆö when invoked from the window header, targets the current open window by default but offers the most-recently-closed window as an alternative radio. When invoked from a past session page, pre-targets that session.

**Surface card states** (copy is counts-only ŌĆö never theme content):
- First-time / therapist exists, no themes yet: "Your therapy prep is ready to start building. Open Therapy."
- Open window, N themes, within typical cadence: quiet card near the bottom. "Therapy prep ┬Ę N themes."
- `openedAt` older than typical cadence (see below): promoted higher. "Your next session is probably coming up. N themes queued."
- Ōēź1 starred theme with `carriedForwardCount >= 2`: promoted. "A few threads keep carrying forward. Worth a look."
- PIN locked: card hidden entirely.

**Typical cadence** is learned from the median duration of the user's last ~4 closed windows. Before that history exists, it defaults to 7 days.

### Data flow

1. User opens `/therapy`. PIN gate; on unlock, `useTherapist()` subscribes.
2. Zero therapists ŌåÆ `TherapistSetupForm`. On submit: create `therapists/{id}` and the first `therapy_windows/{id}` with `status: 'open'`. Trigger `regenerateTherapyWindow` via callable (one-shot ŌĆö the first window has no sources yet; result is usually "nothing to cluster").
3. Otherwise ŌåÆ `useTherapyWindow(therapistId)` subscribes to the open window, its themes (via `where('windowId', '==', openWindow.id)`), and its notes. Renders workspace.
4. User taps star / dismiss / sets a note ŌåÆ `useTherapyActions` writes `userState` on the theme doc. Rules allow.
5. User taps Refresh ŌåÆ client calls the callable; function re-runs generation; themes rewrite in place, preserving `userState`.
6. User taps Import notes ŌåÆ modal ŌåÆ write `therapy_notes/{id}`, push id onto `window.noteIds`.
7. User taps I had a session ŌåÆ `SessionCloseSheet` ŌåÆ on Confirm, client calls `closeTherapyWindow({ windowId, sessionDate, discussedThemeIds, transcript? })`. The function performs the batch (theme `discussedAt` / `windowId` / `lifecycle.carriedForwardCount` updates, window status transition, optional note creation, new window creation) and synchronously runs regen on the new window. Client receives the completed new-window id and re-subscribes.
8. Daily cron picks up any open window with stale `lastRegeneratedAt` and runs regen server-side. User's next visit sees fresh output.
9. The Surface subscribes to a minimal aggregate view (open window exists? theme count? any `carriedForwardCount >= 2`?). No theme content crosses into The Surface.

### Extensibility to couples / family

The `therapists.kind` field is the hinge. v1 only writes `'individual'`. When `'couples'` or `'family'` ships later, forward changes required:

- `therapists` gains `memberUserIds: string[]` alongside (or replacing) `ownerUserId`. Rules grant read/write to any member.
- `regenerateTherapyWindow` gathers source material across all members' journals, respecting entry visibility (only entries whose `sharedWithUserIds` covers the whole group are included).
- Workspace renders collaborative state: starring shows who starred; per-theme notes carry authorship.
- Session close requires either single-initiator (trust model) or one-confirms-other.
- PIN gating becomes per-member (each member's own PIN gates their access).

None of this runs in v1. The point of calling it out: the core loop (windows, themes, carry-forward, notes) is identical across kinds, so the forward path is a schema widening plus rule/function rewrites, not a redesign.

## Testing

- **`regenerateTherapyWindow` unit tests (functions)** ŌĆö input gathering scope (time window, visibility), identity-matching path preserves `userState`, new-theme path populates `lifecycle.firstSeenWindowId`, soft-retire path drops from `themeIds` without deleting the doc, malformed LLM output leaves window untouched.
- **Firestore rules tests** ŌĆö owner can CRUD on all four collections; non-owner denied reads; `therapy_themes.lifecycle` and `sourceRefs` are not client-writable; `therapy_windows.status` cannot transition closed ŌåÆ open.
- **`closeTherapyWindow` unit tests (functions)** ŌĆö carried-forward themes get `windowId` reassigned and `carriedForwardCount` incremented; discussed themes get `discussedAt` set; closed window has `status: 'closed'` and `closedAt` set; new window is created with correct `carriedForwardFromWindowId`; rejects close-twice; transcript writes a `therapy_notes` attached to the closing window.
- **`useTherapyActions.test.ts`** ŌĆö `toggleStar` / `toggleDismiss` / `setNote` write to `userState` only; `closeSession()` invokes the callable with the correct payload; `importNote()` creates a `therapy_notes` doc and pushes onto `window.noteIds`.
- **`ThemeCard.test.tsx`** ŌĆö renders `carried` pip when `carriedForwardCount > 0`; star and dismiss toggle state; source ref row expands; dismissed cards hidden by default.
- **`SessionCloseSheet.test.tsx`** ŌĆö default all unchecked; select-all works; empty-selection confirmation still closes window; paste-transcript creates `therapy_notes`.
- **`SurfaceTherapyCard.test.tsx`** ŌĆö renders nothing while PIN-locked; renders counts-only copy when unlocked; never renders theme titles.
- **E2E** ŌĆö full authenticated flow is blocked by the outstanding Playwright auth-fixture work (tracked in project memory as `project_e2e_auth_fixture_needed`). Integration stays component-level until that fixture exists; noted here so it is not silently skipped.

## Rollout order (for the implementation plan)

1. **Schema + rules + indexes.** Add `src/types/therapy.ts`, update `firestore.rules`, add composite indexes. Deploy rules and indexes before client code so the client's first read paths exist.
2. **PIN-gated `/therapy` shell + therapist setup.** Empty state, create-therapist form, `useTherapist` hook. No regen yet.
3. **`regenerateTherapyWindow` Cloud Function.** Input gathering, prompt, identity matching, write-phase batch. Tested against seeded data before UI depends on it.
4. **Workspace UI.** `WindowHeader`, `ThemeCard`, `ThemeList`, `useTherapyWindow`, `useTherapyActions` (star / dismiss / note / refresh).
5. **Session-close flow.** `closeTherapyWindow` Cloud Function + `SessionCloseSheet` client UI + synchronous regen on the new window.
6. **Note import.** `ImportNotesModal` + `ClosedSessionPage` with transcript rendering.
7. **Surface card.** `SurfaceTherapyCard` + the minimal aggregate subscription. PIN-lock-aware.
8. **Nav integration.** Third book link in `SpreadHome` and header. Deliberately last ŌĆö the feature is stable before it is discoverable.

## Risks / open questions

- **LLM theme identity matching is fragile.** If the model fails to mark "this new theme updates existing theme X" correctly, starred/dismissed state gets stranded on soft-retired docs. Mitigation: always preserve existing `userState`; the soft-retired docs keep their state and can be surfaced in a "previously seen" bucket if matching breaks in practice. Worth monitoring during dogfood.
- **Transcripts near the 1MB Firestore doc cap.** A ~200k-char paste hits the limit. Mitigation: check length on save (warn at ~180k chars); defer multi-doc splitting until it actually happens.
- **Cron reliability.** A missed scheduled run ŌåÆ stale workspace. Low stakes (user can refresh manually), but worth logging successful/failed runs to catch silent failures.
- **"Typical cadence" heuristic.** With only 1ŌĆō2 closed windows, the 7-day default governs. Not a problem, just flagging that early Surface promotion timing will feel arbitrary until a few sessions have accrued.
- **Naming.** Spec uses "Therapy" as the book name. If future expansions (coaching, spiritual direction, other forms of guided reflection) want the same primitive, a broader name might fit ŌĆö deferring that choice; "Therapy" is fine for v1.
